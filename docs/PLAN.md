## 1. 비전 & 목표

**한 문장:** 과일 위탁판매 셀러의 반복 업무(소싱·발주·상세페이지·CS)를 자동화해,
**"무엇을 팔지 → 어떻게 팔지 → 운영"** 전 과정을 한 도구에서 끝낸다.

- 1차 목표: **나(운영자) 한 사람의 위탁판매를 자동화**하는 개인 도구
- 2차 목표(추후): 다른 셀러도 쓸 수 있는 서비스(SaaS)로 확장 가능한 구조 유지

---

## 2. 핵심 결정 (2026-06-18 확정)

| 항목 | 결정 | 함의 |
|---|---|---|
| 제품 성격 | **내 판매 자동화 우선** (단일 사용자) | 초기엔 **로그인/다중사용자/결제 불필요** → 빠른 구현. 단, 데이터 구조는 추후 멀티테넌트 확장 가능하게. |
| 판매 채널 | **쿠팡 우선**, 추후 확장(스마트스토어 등) | 발주/상세페이지/CS 자동화는 쿠팡 기준 우선 설계 |
| 키워드 리서치 소스 | **네이버 데이터랩** (검색어트렌드 + 쇼핑인사이트) | 이미 연동 완료. 쿠팡엔 동등한 트렌드 API가 약해서 리서치는 네이버로. |
| 키워드 등급 기준 | **경쟁 대비 기회** (검색량 ÷ 경쟁상품수) 지향 | 검색량은 네이버, **경쟁상품수는 크롤링(1순위) → 안 되면 네이버 신호로 폴백**. 쿠팡 판매자 API로는 시장 경쟁수 불가(내 상품만 조회). |
| 쿠팡 API 활용 | **판매자(Wing) API = E2·E3·E4 엔진** | 주문/상품등록/CS 자동화에 사용. 경쟁수 측정용은 아님. |
| 1순위 개발 | **E1 키워드/소싱 고도화** (첫 화면: 키워드 분석기) | 이미 있는 기능을 키우는 것이라 리스크 최저 |

## 3. 타겟 사용자 / 페르소나

- **주 사용자:** 과일 위탁판매를 하는 1인 셀러(=운영자 본인). 비개발자.
- **니즈:** "지금 뭘 올려야 잘 팔리나"를 빠르게 판단 + 주문/발주/CS 손품 줄이기.
- **사용 맥락:** 주로 데스크톱, 가끔 모바일로 트렌드 확인.

---

## 4. 제품 범위 — 5대 축

1. **상품 소싱 (키워드/트렌드 리서치)** ← 현재 위치, E1
2. **판매 계획** (무엇을 언제 얼마에 — 피크 예측 기반)
3. **자동 발주** (주문내역 → 도매처 발주서)
4. **상세페이지 자동화** (상품정보 → 이미지/문구)
5. **자동 CS** (문의/리뷰 응대)

## 5. 로드맵 — Epic

각 Epic은 **수직 슬라이스**(UI+API+데이터)로 구현하고, 끝나면 라이브 배포.

### 🟢 E1 — 키워드/소싱 고도화 (지금)
영상의 "키워드 모음집/선정" 화면이 목표. **첫 산출물 = 키워드 분석기.**

**E1-1. 키워드 분석기 (먼저 구현)** — 과일/키워드 입력 → 한 화면에:
> 출시는 2단계 분리(확정): **1a = 네이버 데이터만으로 분석기 출시**(추이·피크·D-day·세분류 비교·임시 등급) → **1b = 쿠팡 경쟁상품수 크롤링 붙여 '경쟁 대비 기회' 등급 고도화.** 크롤링이 막혀도 1a는 독립적으로 가치 제공.
- [ ] **2년 검색 추이 차트** (네이버 검색어트렌드, 좌=과거→우=최근)
- [ ] **시즌 피크 예측** — 예상 피크 검색량, 피크월, **D-day**, 작년/재작년 피크 비교
- [ ] **등급(S/A/B…)** — 경쟁 대비 기회 점수 (검색량 ÷ 경쟁상품수)
  - 경쟁상품수: **쿠팡 검색결과 크롤링(1순위)**, 실패 시 **검색량·시즌·급등세 신호로 폴백**
- [ ] **세분류 vs 대표키워드 비교** ("청송사과 vs 사과" — 세분류가 더 높으면 우량 신호)
- [ ] 쿠팡 검색 바로가기 버튼

**E1-2. 그다음 순차 구현**
- [ ] **키워드 모음집(그리드)** — 등급/현재/예상피크/D-day 카드 일람 + 알람
- [ ] **세분류/롱테일 발굴** (자동완성·연관검색어)
- [ ] **실데이터 기반 "이번 달 추천"** (현재 샘플 → 실데이터 전환)
- [ ] **인기검색어 TOP 리스트** (네이버 쇼핑인사이트 — API 제공 범위 조사 후)

- **열린 질문 / 조사:**
  - 쿠팡 경쟁상품수: 크롤링 안정성·약관 / 대안으로 **쿠팡 파트너스 검색 API** 가능성 (→ 아키텍처 창 조사)
  - 네이버 쇼핑인사이트 "인기검색어 TOP500" API 제공 범위
  - 등급 점수식의 가중치 (검색량 vs 경쟁 vs 시즌)

### ⚪ E2 — 자동 발주 시스템  (쿠팡 주문 API 활용)
- [ ] **쿠팡 주문(OrderSheet) API로 신규 주문 자동 수집** (엑셀 수기 대체)
- [ ] 상품 ↔ 도매처 매핑 테이블 (저장 필요 → 이때 경량 DB 도입)
- [ ] 도매처별 발주서 자동 생성/다운로드
- [ ] (선택) 쿠팡 배송 API로 **송장 자동 업로드**
- **열린 질문:** 도매처 발주 포맷 표준화 범위, 쿠팡 주문 API 권한/한도

### ⚪ E3 — 상세페이지 + 상품등록 자동화  (쿠팡 상품 API 활용)
- [ ] 상품 정보 입력 → 상세페이지 문구/구성 자동 생성 (AI 카피 + 템플릿)
- [ ] 이미지 템플릿/조합
- [ ] **쿠팡 상품(Product) API로 자동 등록** (카테고리 조회 → 생성)
- **열린 질문:** 자동 vs 반자동(검수), 카테고리 매핑 정확도

### ⚪ E4 — 자동 CS 관리  (쿠팡 CS/반품 API 활용)
- [ ] **쿠팡 고객문의·반품·교환 API 조회** → 분류
- [ ] 응대 템플릿 추천/자동 답변
- **열린 질문:** 자동 답변 허용 범위(완전자동 vs 추천)
